# Code Generation

When you run `radiant generate` (or `radiant dev`), the compiler processes `.radiant` files and produces three artifacts: a JSON schema, a TypeScript runtime entry point, and TypeScript type definitions. This page explains what each file contains and how to use it.

## Output Files

| File | Location | Purpose |
|---|---|---|
| `schema.json` | `radiant/runtime/schema.json` | The compiled schema as JSON (used by the runtime) |
| `runtime.ts` | `radiant/runtime/runtime.ts` | The runtime entry point with `createRadiant()` factory |
| `radiant-types.ts` | Project root | TypeScript interfaces for all collections and globals |

The output directory for `schema.json` and `runtime.ts` can be customised via the `output` config key:

```radiant
config {
  output: "../src/generated"
}
```

## `schema.json`

The compiled schema is a JSON object containing all collections, globals, and config settings. Internal compiler metadata (tokens, URIs) is stripped before serialisation.

```json
{
  "collections": [
    {
      "slug": "users",
      "fields": [
        { "name": "name", "type": "text" },
        { "name": "email", "type": "email", "unique": true },
        { "name": "password", "type": "password" },
        { "name": "role", "type": "enum", "values": ["admin", "user"], "default": "user" }
      ],
      "auth": true
    },
    {
      "slug": "todos",
      "fields": [
        { "name": "title", "type": "text" },
        { "name": "completed", "type": "boolean", "default": false },
        { "name": "author", "type": "relationship", "target": "users" }
      ]
    }
  ],
  "globals": [],
  "core": { "api": { "prefix": "/api" } },
  "security": { "auth": { "strategies": ["jwt"] } }
}
```

This file is the single source of truth for the runtime. The `runtime.ts` file embeds it directly.

## `runtime.ts`

The runtime entry point imports `RadiantRuntime` from the runtime package and exports a `createRadiant()` factory:

```typescript
// runtime.ts (Auto-generated)
// Do not edit this file directly.

import { RadiantRuntime, type RadiantConfig } from '@codesordinatestudio/radiant-bun';
import type { Collections } from '../radiant-types';

// 1. Internal AST
const internalSchema = {
  "collections": [ ... ],
  "security": { ... }
} as const;

// 2. Radiant SDK Factory
export function createRadiant(config: RadiantConfig) {
  return new RadiantRuntime<Collections>(internalSchema as any, config);
}
```

You use this in `src/app.ts`:

```typescript
// src/app.ts
import { createRadiant } from "../radiant/runtime";
import { sqlite } from "@codesordinatestudio/radiant-plugin-sqlite";

export const app = createRadiant({
  adapter: sqlite({ url: process.env.DATABASE_URL! }),
});
```

The `createRadiant()` function accepts a `RadiantConfig` object with:
- `adapter` — a database adapter instance (SQLite, PostgreSQL, MongoDB, Redis, SurrealDB)
- `email` — optional email transport plugin
- Additional runtime configuration

## `radiant-types.ts`

The TypeScript types file provides fully typed interfaces for every collection and global. This is what gives you compile-time safety in your hooks, access rules, and custom routes.

### 1. Core Models

```typescript
export interface Users {
  id: string;
  name: string;
  role: "admin" | "user";
}

export interface Todos {
  id: string;
  title: string;
  completed: boolean;
  author: string;  // relationship is string in base model
}
```

Note that `password` fields are omitted from the model interface — they are never returned in API responses.

### 1b. Populated Models

Populated models resolve `relationship` fields to their full referenced type:

```typescript
export interface TodosPopulated {
  id: string;
  title: string;
  completed: boolean;
  author: Users;  // fully populated user object
}
```

### 2. Create / Update Inputs

```typescript
export interface UsersCreate {
  name: string;
  role?: "admin" | "user";  // optional because it has @default
}

export type UsersUpdate = Partial<UsersCreate>;

export interface TodosCreate {
  title: string;
  completed?: boolean;  // optional because it has @default(false)
  author: string;
}

export type TodosUpdate = Partial<TodosCreate>;
```

Fields with `@default` or `@optional` are optional in the `Create` input. `Update` types are always `Partial<Create>`.

### 3. Query Builder Types

```typescript
export interface TodosWhereClause {
  title?: { eq?: string; neq?: string; like?: string; in?: string[]; nin?: string[]; exists?: boolean };
  completed?: { eq?: boolean; neq?: boolean; in?: boolean[]; nin?: boolean[]; exists?: boolean };
  author?: { eq?: string; neq?: string; in?: string[]; nin?: string[]; exists?: boolean };
  and?: TodosWhereClause[];
  or?: TodosWhereClause[];
}
```

Every field gets a where-clause filter object with operators:
- `eq` — equals
- `neq` — not equals
- `like` — LIKE pattern (string fields)
- `in` — in array
- `nin` — not in array
- `exists` — is not null

Plus `and` / `or` for composing complex queries.

### 4. Collections Registry

```typescript
export type Collections = {
  users: Users;
  todos: Todos;
  __populated: {
    users: UsersPopulated;
    todos: TodosPopulated;
  }
};
```

This type maps collection slugs to their model types. The runtime uses it for type inference when you call `app.collection("todos")`.

## TypeScript Type Mapping

The generator maps Radiant field types to TypeScript types:

| Radiant Type | TypeScript (base) | TypeScript (populated) |
|---|---|---|
| `text`, `textarea`, `email`, `password`, `date` | `string` | `string` |
| `boolean` | `boolean` | `boolean` |
| `integer`, `number` | `number` | `number` |
| `select("a", "b")` | `"a" \| "b"` | `"a" \| "b"` |
| `["a", "b"]` (enum) | `"a" \| "b"` | `"a" \| "b"` |
| `relationship("x")` | `string` | `X` (the referenced model) |
| `multiselect(...)` | `string[]` | `string[]` |
| `json`, `richtext`, `upload` | `any` | `any` |
| `array` / `[]` suffix | `T[]` | `T[]` |
| nested object | `{ ... }` | `{ ... }` |

## Using Generated Types

The generated types are imported by your TypeScript source files:

```typescript
// src/hooks.ts
import { app } from "./app";
import type { TodosCreate } from "../radiant-types";

app.hooks("todos", {
  beforeCreate: async (ctx) => {
    // ctx.data is typed as TodosCreate
    if (!ctx.data.title) throw new Error("Title required");
    return ctx.data;
  },
});
```

```typescript
// src/access.ts
import { app } from "./app";

app.access("todos", {
  read: (ctx) => true,
  create: (ctx) => ctx.user?.role === "admin",
  update: (ctx) => ctx.user?.id === ctx.data.author || ctx.user?.role === "admin",
  delete: (ctx) => ctx.user?.role === "admin",
});
```

## Regeneration

Generated files are overwritten on every `radiant generate` or `radiant dev` rebuild. Never edit them by hand — change your `.radiant` files and regenerate.

The generated files include a header comment as a reminder:

```typescript
// runtime.ts (Auto-generated)
// Do not edit this file directly.
```
